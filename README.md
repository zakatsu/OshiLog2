# ①課題名
OshiLog ver2（推し活参戦ログアプリ）

## ②課題内容（どんな作品か）
推し活の現地参戦ログを、タイムラインとマップで振り返れる1ページアプリです。

ver2では、Firebaseの2種類のデータベースを同じアプリ内で使い分けることを目的に実装しました。参戦ログ本体はFirestoreに保存し、タイムラインごとのコメントはRealtime Databaseに保存します。

Firestoreは、参戦ログのように「1件のイベント情報」をドキュメントとして扱い、登録・編集・複製・削除しやすいデータに向いています。Realtime Databaseは、コメントのように追加された内容をすぐ画面へ反映したいツリー型データに向いています。

都市名から緯度経度を判定し、Leafletの地図上にピンとして表示します。タイムラインでログを選択すると、対応する地図ピンへズームします。

なし

## ⑤工夫した点・こだわった点
- Firebaseの使い分け
  - Firestoreに `attendanceLogs` コレクションを作成し、参戦ログをドキュメント単位で保存
  - Realtime Databaseに `comments/{logId}` を作成し、ログごとのコメントを保存
  - Firestoreは構造化されたCRUD、Realtime Databaseは即時反映しやすいコメント欄として役割を分けた

- 機能面
  - 参戦ログの登録・編集・複製・リセットに対応
  - タイムラインごとにコメント欄を開閉し、名前付きコメントを投稿できる
  - 都市名と緯度経度の対応表を `cities.json` で管理
  - コメント数をタイムライン上に表示

- UIへのこだわり
  - タイムラインのレコード選択と地図のピンを連動
  - 選択時にマップが一度引いてから対象ピンへズームする動きにした
  - PCでは登録フォームやコメントフォームを横並びにし、ログ一覧として見やすくした
  - コメント欄はタイムライン行の中に収まるよう、表示幅と1行入力のバランスを調整

## ⑥難しかった点・次回トライしたいこと（又は機能）
- FirestoreとRealtime Databaseでデータ構造・ルール・APIの書き方が違うため、保存先ごとの責務を整理する必要があった
- Realtime Databaseのコメントはリアルタイムに追加される一方、削除済みログに紐づくコメントの扱いも考える必要があった
- 都市名判定は現在 `cities.json` の範囲内なので、次回は住所検索APIやジオコーディングAPIで会場名から自動判定したい
- ログのエクスポート・インポート機能を追加して、バックアップできるようにしたい

## ⑦フリー項目（感想、シェアしたいこと等なんでも）
- ver1のローカル保存型から、ver2ではFirebase保存型へ変更し、端末に依存しないログ管理に近づけました
- FirestoreとRealtime Databaseを同時に使うことで、「どちらもFirebaseのDBだが、得意なデータの持ち方が違う」ことを体感できました
- 参戦ログは後から編集される整理済みデータ、コメントは軽く追加される流動的なデータとして分けると理解しやすかったです
